# LambdaVLIW

`LambdaVLIW` - учебный компилятор Lisp-подобного языка с промежуточным typed SSA и набором вспомогательных интерпретаторов, трансляторов и оптимизационных проходов.

Проект ориентирован на эксперименты с оптимизациями, представлениями данных и обучение общим принципам оптимизирующей компиляции, с расчётом на дальнейшую адаптацию под `e2k`.

`LambdaVLIW` — это набор компонентов:
- парсер Lisp-подобного синтаксиса;
- frontend для языка с `defun`, `let`, `cond`, арифметикой и списочными примитивами;
- SSA как промежуточное представление;
- оптимизатор над SSA;
- C-бэкенд с простым рантаймом и поддержкой пар;
- интерпретаторы промежуточных языков и подмножеств исходного языка

Проект находится в активной разработке


## Основные особенности
- Lisp-подобный синтаксис и функциональный стиль программирования
- Отказ от side-effect конструкций вида `setq` в пользу простоты и оптимизаций
- Собственное typed SSA-представление
- Представление CFG через базовые блоки и список инструкций
- C-бэкенд с встроенным рантаймом и простым управлением памятью

## Цели и мотивация
- Изучить оптимизирующую компиляцию на практике
- Экспериментировать с SSA и оптимизациями на собственном IR
- Исследовать практические вопросы компиляции под VLIW/`e2k`-подобные архитектуры

## Архитектура

### Frontend

Основной frontend расположен в `L_defun/`
Он отвечает за построение SSA-модуля

### Middle-end

Центральное промежуточное представление находится в `SSA/`

В него входят:
- описание SSA-модуля, функций, значений и блоков;
- инструкции в базовых блоках;
- `phi`-узлы;
- типизированные значения (`i1`, `i8`, `i32`, `i64`, `fp32`, `fp64`, `void`);
- дамперы в Graphviz и `L_tri_call` (трёхадресный текстовый язык с скобочным синтаксисом);
- базовый API для мутаций и оптимизатора.

Оптимизации находятся в `optimizer/`
Сейчас уже есть TCO и возможность построить большинство других базовых оптимизаций

### Backend

Основной backend сейчас расположен в `C_backend/`
Он превращает SSA в C-код, который затем можно компилировать системным C-компилятором

Это промежуточный этап, т. к. ассемблер под `e2k` сложный, поэтому я решил сфокусироваться на более общих для компиляторов задачах.

Сейчас есть рабочая fallback-версия, компилирующая в стейт-машину, а нормальная версия с компиляцией в if/while/for находится в разработке

### Сборщик мусора и рантайм

В C-бэкенде встроен минимальный рантайм:
- представление пар как heap-объектов;
- простой список выделенных объектов;
- финальная очистка памяти в `main` через `__gc_cleanup()`.

Это не полноценный GC, но минимальная рабочая инфраструктура для поддержки пар в сгенерированном C

## Поддержка `e2k`
Проект задуман с оглядкой на `e2k` и VLIW-подходы, но текущая поддержка находится на экспериментальной стадии

Планируется:
- полноценный backend, напрямую генерирующий код/IR под `e2k`;
- target-specific планировщик;

VLIW учитывается как целевое направление проекта

## Структура репозитория
- `parser/` — парсер и AST.
- `htable/` — простая хеш-таблица.
- `L_defun/` — frontend исходного языка в SSA.
- `SSA/` — typed SSA, дамперы и тесты.
- `optimizer/` — SSA-оптимизации.
- `C_backend/` — генерация C-кода.
- `compiler/` — основной компиляторный пайплайн.
- `L_interpret/`, `L_int/`, `L_tri_if/`, `L_tri_call/`, `L_var2L_tri_int/` — вспомогательные языки, интерпретаторы и промежуточные шаги.

## Зависимости и требования

Минимальные зависимости:

- `C`-компилятор с поддержкой `C23` / `Posix c`
- `make`
- libc

Полезные дополнительные зависимости:

- `graphviz` для просмотра SSA-графов (`dot`, `xdot`, `kgraphviewer` и т.д.)
- отладочные флаги компилятора (`-g`, `-fsanitize=address` и т.п.) при локальной отладке
- компилятор/тулчейн целевой платформы, если планируется запуск сгенерированного C-кода вне обычного `gcc/clang`-окружения

## Сборка

### Сборка основного компилятора

```bash
make
``` 
в корне соберёт весь проект.
можно вызвать также с целями `debug` / `asan` для соответствующих флагов

главный executable будет `compiler/main`

### Сборка отдельных компонентов
e. g.
```bash
make -C SSA tests
make -C L_tri_call interpret
make -C parser
```

## Компиляция программ

Основной путь сейчас такой:

```bash
Usage: compiler/main [--no-opt] [--dummy-backend|--system-backend] [--dump-dot path.dot] -o output.c input.lisp
  -o <path>         Output C file. Use '-' for stdout.
  --dump-dot <path> Dump SSA Graphviz DOT to file.
  --no-opt          Disable optimizer passes.
  --dummy-backend   Force dummy C backend.
  --system-backend  Force structured backend (default).
```

После этого можно отдельно собрать получившийся `C`-файл системным компилятором^

```bash
cc output.c -o output
./output
```

## Отладка и дампы

### SSA-тесты и дампы

```bash
make -C SSA tests
./SSA/tests
```

Тесты генерируют:

- `.trif` — дамп SSA в `L_tri_call`-подобный формат;
- `.dot` — Graphviz-дамп CFG/SSA.

### Генерация SVG из Graphviz

пример:
```bash
dot -Tsvg SSA/samples/branch_and_phi.dot -o branch_and_phi.svg
```
